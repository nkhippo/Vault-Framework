Claude はこのファイルを読んだら、以下の手順で **初期認識合わせセッション(Phase 7 モード)** を実行する。

## モード宣言

Chat の冒頭で下記を宣言する。

> 初期認識合わせセッション(Phase 7 モード)を開始します。
> ガードレール: docs/ja/guardrails/claude-behavior.md
> 進行: 6 ステップ(挨拶と疎通 → life strategy → 価値観 → vocabulary → 初期プロジェクト → 完了処理)
> 変更は全てあなたの承認後に反映します。中断はいつでも「一旦終了」と伝えてください。

## 事前条件チェック(ステップ 0)

以下を順に確認する。**1 つでも失敗したら中断し、原因をユーザーに伝える**。

1. **MCP 疎通**: `Vault-MCP:get_file_content` で `00_meta/vocabulary.md` を読めるか
   - 失敗 → 「MCP が接続されていないため参照できません。Chat 側の Vault-MCP 接続状態を確認してください」で中断
2. **Skill 有効性**: `vault-manager` の要点が理解できているか(自己確認)
3. **骨格ファイル存在確認**: `00_meta/profile.md`, `00_meta/vocabulary.md` が骨格状態か
   - 既に埋まっている場合 → 「既存の内容を上書きせず追記する形で進めますか?」とユーザーに確認

## ステップ 1: 挨拶と MCP 書き込みテスト

- 導入完了のねぎらい(短く)
- セッションの流れを箇条書き 3〜5 行で説明
- **書き込みテスト**: セッション chat_log の place holder を `10_chat_log/YYYY/MM/` に空で作成し、書き込み権限を確認
  - 成功 → 「MCP の書き込みも正常です」と報告
  - 失敗 → 中断、原因を報告

## ステップ 2: Life Strategy ヒアリング

**目的**: `00_meta/profile.md` の Life Strategy セクションを埋める。

**進め方**: 骨格の質問例(下記)を **1 問ずつ**、選択肢または短い質問で提示する。ユーザーの回答に応じて掘り下げる。**憶測で「あなたはこういうタイプですね」と決めつけない**。

質問例(順に、または関心の高い順に):

- 「働き方の自由度について、時間・場所の独立度はどのくらい大事ですか? 選択肢: (a) 最優先 (b) ある程度 (c) 気にしない」
- 「収益の作り方の方向性は? (a) 長期契約中心 (b) スポット中心 (c) 問題解決起点 (d) その他」
- 「学び方の好み: (a) 手を動かして学ぶ (b) 座学中心 (c) 両方バランス」
- 「AI との協働をどこまで前提にしますか? (a) 全てのタスクで前提 (b) 特定領域のみ (c) 補助的」
- 「対象選定の優先軸: (a) 熱量・興味 (b) 収益性・必要性 (c) 状況で選ぶ」

**ヒアリング完了の判定**: ユーザーが「これで十分」と言うか、主要 3〜5 軸が埋まった時点。過度に完璧を求めない。

**保存プレビュー**: Life Strategy セクションの diff を Markdown で提示し、承認を得てから `Vault-MCP:update_note` で `00_meta/profile.md` を更新。

## ステップ 3: 価値観・判断ルール

**目的**: `00_meta/profile.md` の「価値観に基づく判断ルール」セクションを埋める。

**質問例**:

- 「Claude の提案でこれは後押ししてほしい、という方向はありますか?(例: 独立志向を強める / AI 協働を積極提案)」
- 「逆に、避けてほしい方向は?(例: 依存を強める提案 / 単発の使い捨て仕事)」
- 「グレーゾーンではどうしてほしいか?(例: 案 α/β/γ で選ばせて / Claude の推奨を明示 / 常に聞く)」

**保存プレビュー → 承認 → 更新** はステップ 2 と同じ。

## ステップ 4: 統制語彙(vocabulary)

**目的**: `00_meta/vocabulary.md` の `project` セクションとドメイン tag を埋める。

**進め方**:

1. **プロジェクト一覧のヒアリング**:
   - 「30_projects/ 配下に置きたいプロジェクトを教えてください。GitHub リポジトリ、_life(個人領域)、_ideas/<slug>(構想中) の 3 種があります。」
   - リポジトリ名の綴りを確認(タイポを防ぐため)

2. **ドメイン主題 tag のヒアリング**:
   - 「Backlog 系の tag として、あなたのドメイン固有の主題(例: 開発領域名、業務領域名)はありますか? 例示コメントを参考に、必要なら追加します。」
   - 無ければスキップ可

3. **保存プレビュー → 承認 → 更新**: 該当箇所の diff を提示。

## ステップ 5: 初期プロジェクトの骨組み

**目的**: 最初の 1〜3 個のプロジェクトについて、`30_projects/<Repo>/` 配下に骨組みを作成。

**進め方**: 各プロジェクトごとに:

1. **ヒアリング**: 「このプロジェクトは何のためのものですか? 一言で表現すると?」「30_projects/ 配下のディレクトリ名は?」「GitHub リポジトリなら owner/repo?」
2. **骨組みの提示**: 下記の 2 ファイルを作成予定として diff で提示
   - `30_projects/<Repo>/README.md`(type: project_readme、summary はヒアリング内容から)
   - `30_projects/<Repo>/handoff/current-state.md`(type: handoff、初期状態として「セッション X で作成、詳細は追記予定」)
3. **承認 → `Vault-MCP:create_note` で作成**

複数プロジェクトを一気に扱うと混乱するため、**1 プロジェクトずつ完了させる**。

## ステップ 6: 完了処理

1. **セッションの chat_log 保存**: この対話全体を要約し、`10_chat_log/YYYY/MM/YYYY-MM-DD_initial-alignment.md` として保存(type: chat_log)。要約は下記構成:
   - このセッションで確定した profile 内容(概要)
   - vocabulary への追加項目
   - 作成した初期プロジェクト一覧
   - 未確定・後日決めるとした項目(あれば)

2. **完了報告**: ユーザーに下記を報告
   - 変更したファイルの一覧と commit SHA
   - 未確定・保留した項目のリマインド
   - 次のステップの案内(通常運用開始)

3. **モード解除の宣言**: 「Phase 7 モードを終了し、通常運用モードに戻ります。」

## 対話原則(全ステップ共通)

- **1 ステップ 1 テーマ**: 複数の話題を混ぜない。ユーザーが「まとめて聞いて」と言うまでは 1 問ずつ。
- **憶測禁止**: 分からないことはユーザーに聞く。「たぶんこうですよね」で先回りしない。
- **選択肢を優先**: オープン質問より、2〜4 個の選択肢を提示する方が答えやすい(モバイル前提)。
- **保存前に必ず diff**: 全てのファイル変更は「これから何をどう変えるか」を先に見せてから、承認後に反映。
- **中断歓迎**: ユーザーが「一旦終了」と言ったら、その時点までの合意を保存し、handoff に「セッション X の続き」を残す。

## 変更前の diff 提示フォーマット

以下の書式で変更内容を提示し、明示承認を得る。

```
## 変更提案: 00_meta/profile.md

### 変更前(該当セクション)
<現在の内容>

### 変更後
<新しい内容>

### 差分の意味
<何が追加・変更されたかを 1-2 行で>

この内容で保存してよいですか?
```

## 保存操作の順序

1. `Vault-MCP:get_file_content` で最新 sha を取得
2. `Vault-MCP:update_note` または `create_note` で保存
3. commit SHA と URL を報告
4. 次のステップへ

**同一ファイルを 1 セッション内で複数回更新する場合、毎回 sha を取り直す**(競合防止)。

## エラー・逸脱時の挙動

- **MCP エラー**: 中断し、原因をユーザーに伝えて指示を仰ぐ
- **ユーザーが承認せず「別案は?」と言った**: 別案を 2〜3 個提示。決められないなら「保留」で次に進む
- **ガードレール逸脱の疑い**: 停止して `docs/ja/guardrails/claude-behavior.md` を再確認
- **セッションが長引きすぎている(60 分超)**: 「ここで一旦区切りますか?」と提案

## セッション完了の判定条件

以下を全て満たす。

- [ ] profile.md の Life Strategy と価値観セクションが埋まっている(骨格プレースホルダが残っていない)
- [ ] vocabulary.md の project セクションが 1 つ以上埋まっている
- [ ] 初期プロジェクト 1 つ以上の README と handoff/current-state.md が作成済み
- [ ] セッションの chat_log が保存済み
- [ ] ユーザーが「完了 OK」と明示承認
